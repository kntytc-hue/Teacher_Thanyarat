    Unit 15: Course Summary & Capstone Integration

    หน่วยที่ 15 สรุปรายวิชา

    1. สรุปภาพรวมรายวิชา (Course Overview Summary)

    วิชา **31909-1001 ระบบปฏิบัติการ (Operating Systems)** มุ่งเน้นการศึกษาสถาปัตยกรรม กลไกการทำงาน และการบริหารจัดการทรัพยากรของระบบคอมพิวเตอร์อย่างเป็นระบบ โดยระบบปฏิบัติการทำหน้าที่เป็นตัวกลาง (Intermediary) ระหว่างผู้ใช้/โปรแกรมประยุกต์ และฮาร์ดแวร์จริง ตลอดทั้งภาคเรียน ผู้เรียนได้ศึกษาครอบคลุมมิติสำคัญ 6 ด้านหลัก ได้แก่:

    การบริหารโปรเซสและสเตรด

    การจัดการสถานะโปรเซส, สเก็ตจูลิง CPU, การสลับบริบท (Context Switch), และโครงสร้างแบบ Threads ในระบบมัลติคอร์

    การประมวลผลขนานและเดดล็อก

    ปัญหา Race Condition, กลไกการประสานเวลา (Semaphore, Mutex) และแนวทางจัดการ Deadlock (Banker's Algorithm)

    หน่วยความจำหลักและเสมือน

    การแบ่งหน้า (Paging), การแบ่งส่วน (Segmentation), Demand Paging, Page Fault Handling และอัลกอริทึมแทนที่หน้า (FIFO, LRU, OPT)

    ระบบไฟล์และดิสก์

    โครงสร้างไดเรกทอรี, การจัดสรรพื้นที่ดิสก์ (Contiguous, Linked, Indexed), และ Disk Scheduling (FCFS, SSTF, SCAN, C-SCAN)

    ความปลอดภัยและการปกป้อง

    การพิสูจน์ตัวตน, การควบคุมการเข้าถึง (Access Control Matrix/ACL), มัลแวร์, และมาตรการรักษาความปลอดภัยระดับเคอร์เนล

    การจำลองระบบและคลาวด์

    แนวคิด Virtualization, Hypervisor (Type 1 & Type 2), Containerization (Docker), และสถาปัตยกรรมระบบคลาวด์ยุคใหม่

    2. ทบทวนแนวคิดสำคัญ (Review of Key Concepts)

    การสรุปองค์ความรู้ที่สำคัญจากการเรียนตลอดทั้งภาคเรียนเพื่อให้เห็นภาพรวมและจุดเชื่อมโยงของแต่ละทฤษฎี:

    Process คือโปรแกรมที่กำลังประมวลผลในหน่วยความจำ โดยมีโครงสร้าง PCB (Process Control Block) เก็บสถานะต่างๆ สเก็ตจูเลอร์ทำหน้าที่จัดสรร CPU ให้โปรเซสด้วยอัลกอริทึม เช่น **FCFS** (มาก่อนได้ก่อน เกิด Convoy Effect), **SJF/SRTF** (ได้เวลาเฉลี่ยดีที่สุดแต่ต้องรู้อนาคต), **Priority** (เสี่ยง Starvation แก้ด้วย Aging) และ **Round Robin (RR)** (เหมาะสมกับ Time-sharing ขึ้นกับขนาดของ Time Quantum)

    เมื่อโปรเซสทำงานพร้อมกันและเข้าถึง Critical Section อาจเกิด Race Condition ต้องแก้ด้วยการใช้อุปกรณ์ประสานเวลา เช่น Mutex Lock และ Semaphore ส่วน Deadlock เกิดขึ้นเมื่อครบเงื่อนไข 4 ประการ (Mutual Exclusion, Hold and Wait, No Preemption, Circular Wait) สามารถหลีกเลี่ยงได้โดยใช้ **Banker's Algorithm** หรือใช้ Resource Allocation Graph ในการตรวจจับ

    Paging ช่วยแก้ปัญหา External Fragmentation โดยแปลง Logical Address เป็น Physical Address ผ่าน Page Table ร่วมกับ TLB สำหรับ Virtual Memory ใช้ Demand Paging ในการโหลดหน้าเข้ามาเมื่อต้องการใช้งานจริง หาก RAM เต็มจะเกิด Page Replacement ด้วยอัลกอริทึม **FIFO** (อาจเกิด Belady's Anomaly), **OPT** (เชิงอุดมคติ) หรือ **LRU** (ใช้งานจริงดีที่สุด) และต้องระวังการเกิด Thrashing

    การจัดสรรพื้นที่ไฟล์มี 3 แบบหลัก (Contiguous, Linked, Indexed เช่น Inode ใน Unix) การบริหารหัวอ่านดิสก์ใช้ Disk Scheduling เช่น **SSTF** (เลือกหัวอ่านใกล้นิ้วที่สุด เสี่ยง Starvation), **SCAN** (อัลกอริทึม ลิฟต์) และ **C-SCAN** (กวาดทางเดียวสม่ำเสมอ) เพื่อลดระยะเวลา Seek Time

    3. โครงงานประจำวิชา (Course Project)

    **โครงงานระบบจำลองการทำงานของระบบปฏิบัติการ (Operating System Simulation Suite)** เป็นโครงงานกลุ่มเพื่อประยุกต์ทฤษฎีสู่การพัฒนาซอฟต์แวร์จริง นักศึกษาต้องเลือกพัฒนาแบบจำลองตามข้อกำหนดดังนี้:

    ข้อกำหนดหลักของโครงงาน (Project Core Specifications)
    • ขอบเขตผลงาน: สร้าง Web Application หรือ Desktop Application ที่สามารถจำลองการทำงานได้อย่างน้อย 2 โมดูลหลัก (เช่น CPU Scheduling Simulator ร่วมกับ Page Replacement Simulator)
    • ความสามารถด้าน Interactive Visualization: ต้องแสดงขั้นตอนการประมวลผลแบบ Step-by-Step มี Gantt Chart สำหรับ CPU Scheduling หรือ Memory Map/State Diagram สำหรับ Virtual Memory
    • การเปรียบเทียบประสิทธิภาพ (Performance Analysis): คำนวณค่าสถานะ เช่น Average Waiting Time, Turnaround Time, Page Fault Count หรือ Disk Seek Distance เปรียบเทียบอัลกอริทึมต่างๆ พร้อมกราฟสรุปผล
    • สิ่งส่งมอบ (Deliverables): ซอร์สโค้ดฉบับสมบูรณ์บน GitHub/GitLab, เล่มรายงานโครงงาน (Project Documentation), และสไลด์พร้อมคลิปวิดีโอสาธิตการทำงาน

    4. การมอบหมายงาน (Assignment Summary)

    การประเมินผลระหว่างเรียนครอบคลุมภาระงานแบบบุคคลและแบบกลุ่มตลอด 15 หน่วยการเรียนรู้ เพื่อสะสมคะแนนทักษะปฏิบัติ ดังสรุปในตาราง:

    รหัสงาน ชื่อภาระงาน / หัวข้อการฝึกปฏิบัติ ลักษณะงาน สัดส่วนคะแนน
    LAB-01 การใช้งาน Shell Commands & Bash Scripting เบื้องต้น งานบุคคล 5%
    LAB-02 การคำนวณเปรียบเทียบ CPU Scheduling Algorithms (FCFS, SJF, RR) งานบุคคล 5%
    LAB-03 การแก้ปัญหา Producer-Consumer และ Synchronization ด้วย Semaphore งานคู่ 5%
    LAB-04 การจำลองคำนวณ Banker's Algorithm และ Page Replacement Tracing งานบุคคล 5%
    PRJ-01 โครงงานจำลองระบบปฏิบัติการ (OS Simulator Project) งานกลุ่ม 20%

    5. แฟ้มสะสมงาน (Portfolio Guidance)

    แฟ้มสะสมงาน (Portfolio) เป็นเอกสารรวบรวมชิ้นงาน รหัสพัฒนา และบันทึกการเรียนรู้ตลอดภาคเรียนเพื่อใช้เป็นหลักฐานแสดงสมรรถนะวิชาชีพ โครงสร้างแฟ้มสะสมงานดิจิทัลควรประกอบด้วย:

    1. ส่วนประวัติและผลการเรียน (Profile & Track)

    ข้อมูลส่วนตัว, ประวัติตามสายวิชาชีพ, สรุปผลการทำแบบทดสอบประจำหน่วย (Unit Quiz Records)

    2. รวบรวมงานแล็บและซอร์สโค้ด (Lab Artifacts)

    ลิงก์แนบ GitHub Repository รวมโค้ดการทดลองพร้อมภาพบันทึกหน้าจอการทำงาน (Execution Screenshots)

    3. เอกสารโครงงานประจำวิชา (Capstone Project)

    คู่มือการใช้งาน, สถาปัตยกรรมระบบ (Architecture Diagram), และวิดีโอการนำเสนอโครงงาน

    4. บันทึกการสะท้อนคิด (Self-Reflection Log)

    บันทึกการเรียนรู้ ปัญหาที่พบระหว่างการทดลอง วิธีการแก้ไข และเป้าหมายการพัฒนาต่อยอด

    6. การสอบปลายภาค (Final Exam Structure)

    การประเมินผลการสอบปลายภาคเรียน (สัดส่วนคะแนน 30%) มีวัตถุประสงค์เพื่อวัดความรู้ความเข้าใจเชิงทฤษฎีและทักษะการคำนวณแก้ปัญหาทางสถาปัตยกรรมระบบ โดยมีโครงสร้างข้อสอบดังนี้:

    คะแนนเต็ม 100 คะแนน
    (คิดเป็น 30% ของรายวิชา)
    • ตอนที่ 1: แบบเลือกตอบ (Multiple Choice) — จำนวน 30 ข้อ (30 คะแนน) ครอบคลุมนิยามและแนวคิดพื้นฐานหน่วยที่ 1-14
    • ตอนที่ 2: แบบเขียนตอบแนวคิด (Short Answer) — จำนวน 4 ข้อ (20 คะแนน) อธิบายหลักการทำงานและกลไกของระบบปฏิบัติการ
    • ตอนที่ 3: แบบวิเคราะห์คำนวณ (Problem Solving) — จำนวน 3 ข้อใหญ่ (50 คะแนน) แสดงวิธีทำคำนวณ CPU Scheduling, Banker's Algorithm และ Page Replacement

    7. การสะท้อนคิดการเรียนรู้ (Reflection Framework)

    การสะท้อนคิด (Reflection) เป็นกระบวนการสำคัญในการแปลงประสบการณ์การเรียนรู้ไปสู่ความเข้าใจที่ลึกซึ้ง นักศึกษาควรทบทวนและประเมินตนเองตามกรอบแนวคิด 3D (Discover, Difficulties, Development):

    1. Discover (สิ่งที่ค้นพบ)

    เข้าใจการทำงานเบื้องหลังเบื้องหลังระบบคอมพิวเตอร์ ว่าการจัดการทรัพยากร CPU, RAM, Disk มีอัลกอริทึมในการจัดสรรเพื่อให้เกิดประสิทธิภาพสูงสุดได้อย่างไร

    2. Difficulties (อุปสรรค)

    ความท้าทายในหัวข้อคำนวณขั้นสูง เช่น สภาวะการแข่งขัน (Race Condition), การจำลองอัลกอริทึมการแทนที่หน้า และการทำความเข้าใจสถานะปลอดภัยใน Banker's Algorithm

    3. Development (การพัฒนา)

    การนำความรู้การจัดสรรทรัพยากรและการจัดการระบบไปประยุกต์ใช้ในการเขียนโปรแกรมที่มีประสิทธิภาพ การปรับแต่งคอนฟิกเซิร์ฟเวอร์ และการวิเคราะห์ระบบ

    8. แนวข้อสอบปลายภาค (Final Exam Guidelines & Sample Questions)

    ตัวอย่างโจทย์คำนวณเน้นย้ำสำหรับการเตรียมตัวสอบปลายภาค พร้อมขั้นตอนการวิเคราะห์เฉลยแบบเป็นลำดับขั้นตอน:

    ตัวอย่างโจทย์ที่ 1: การคำนวณสเก็ตจูลิง CPU แบบ Round Robin (Quantum = 4)

    กำหนดโปรเซส P1, P2, P3 มีเวลาประมวลผล (Burst Time) ดังนี้ P1 = 10ms, P2 = 3ms, P3 = 7ms (เดินทางมาถึงเวลา 0 พร้อมกัน)

    ลำดับการทำงานใน Gantt Chart:
    |-- P1 (0-4) --|-- P2 (4-7) --|-- P3 (7-11) --|-- P1 (11-15) --|-- P3 (15-18) --|-- P1 (18-20) --|

    การคำนวณเวลารอคอย (Waiting Time):
    - P2 จบที่เวลา 7 -> Waiting Time = 4 - 0 = 4 ms
    - P3 จบที่เวลา 18 (รันช่วง 7-11 และ 15-18) -> Waiting Time = (7-0) + (15-11) = 7 + 4 = 11 ms
    - P1 จบที่เวลา 20 (รันช่วง 0-4, 11-15, 18-20) -> Waiting Time = 0 + (11-4) + (18-15) = 0 + 7 + 3 = 10 ms
    Average Waiting Time = (10 + 4 + 11) / 3 = 25 / 3 = 8.33 ms
    ตัวอย่างโจทย์ที่ 2: การคำนวณ Page Replacement ด้วย LRU (ขนาดเฟรม = 3)

    ลำดับการเรียกหน้า (Page Reference String): 1, 2, 3, 4, 1, 2, 5, 1, 2, 3

    - หน้า 1 -> [1, -, -] (Fault 1)
    - หน้า 2 -> [1, 2, -] (Fault 2)
    - หน้า 3 -> [1, 2, 3] (Fault 3)
    - หน้า 4 -> เต็ม! เตะหน้า 1 (ใช้ย้อนหลังเก่านานสุด) -> [4, 2, 3] (Fault 4)
    - หน้า 1 -> เต็ม! เตะหน้า 2 -> [4, 1, 3] (Fault 5)
    - หน้า 2 -> เต็ม! เตะหน้า 3 -> [4, 1, 2] (Fault 6)
    - หน้า 5 -> เต็ม! เตะหน้า 4 -> [5, 1, 2] (Fault 7)
    - หน้า 1 -> มีใน RAM อยู่แล้ว -> [5, 1, 2] (Hit!)
    - หน้า 2 -> มีใน RAM อยู่แล้ว -> [5, 1, 2] (Hit!)
    - หน้า 3 -> เต็ม! เตะหน้า 5 -> [3, 1, 2] (Fault 8)
    สรุปจำนวน Page Fault ทั้งหมด = 8 ครั้ง

    9. แนวทางการนำความรู้ไปใช้จริง (Real-World Application)

    องค์ความรู้จากวิชาระบบปฏิบัติการถือเป็นรากฐานสำคัญในการต่อยอดสู่สายงานอาชีพเทคโนโลยีชั้นนำ ตัวอย่างการประยุกต์ใช้งานในอุตสาหกรรม:

    Systems & Cloud Engineer

    เข้าใจการบริหาร CPU affinity, Memory Caching และ Kernel Tuning เพื่อปรับแต่งเซิร์ฟเวอร์ Linux/Windows ให้รองรับ Workload ขนาดใหญ่ในระบบ Cloud

    DevOps & Container Specialist

    ประยุกต์ใช้ Linux Namespaces และ Cgroups (พื้นฐานโครงสร้าง OS Isolation) ในการจัดการ Docker Containers และ Kubernetes Clusters

    Backend & Systems Software Developer

    พัฒนาโปรแกรมที่รองรับ Concurrent Multi-threading ป้องกันปัญหา Race Condition และการเข้าถึงไฟล์ร่วมกันได้อย่างปลอดภัยและรวดเร็ว

    Cybersecurity & Security Analyst

    เข้าใจสถาปัตยกรรม Kernel vs User Space, Memory Corruption (Buffer Overflow) และการควบคุมสิทธิ์ Access Control เพื่อป้องกันช่องโหว่ของระบบ

    10. Rubric การประเมินโครงงาน (Project Assessment Rubric)

    เกณฑ์การเกรดและประเมินผลคะแนนโครงงานประจำวิชา (คะแนนเต็ม 20 คะแนน) แบ่งออกเป็น 5 เกณฑ์หลัก ดังรายละเอียดต่อไปนี้:

    เกณฑ์การประเมิน ดีมาก (4 คะแนน) ดี (3 คะแนน) พอใช้ (2 คะแนน) ปรับปรุง (1 คะแนน)
    1. ความถูกต้องทางอัลกอริทึม (Algorithm Accuracy) อัลกอริทึมทำงานถูกต้อง 100% ครอบคลุมทุก Edge Case และแสดงผลถูกต้องตามทฤษฎี อัลกอริทึมทำงานถูกต้องเป็นส่วนใหญ่ มีข้อผิดพลาดเล็กน้อยในกรณีซับซ้อน คำนวณผลลัพธ์ถูกต้องบางส่วน มีจุดผิดพลาดเชิงตรรกะในหลายกรณี ผลลัพธ์คำนวณไม่ถูกต้อง หรือซอฟต์แวร์เกิด Crash ระหว่างประมวลผล
    2. การออกแบบส่วนต่อประสานและ Animation (UI/UX) การแสดงผล Visualisation ชัดเจน เข้าใจง่าย สวยงาม มี Interactive ควบคุม Step-by-Step ได้ดีเยี่ยม หน้าจอเป็นระเบียบ สวยงาม มี Animation หรือภาพประกอบแสดงขั้นตอนครบถ้วน แสดงผลเป็นข้อความตารางพื้นฐาน ขาดความสวยงามหรือ Interactive animation หน้าจอใช้งานยาก ไม่มีการจัดรูปแบบ ขาดการแสดงผลเชิงภาพ
    3. คุณภาพโค้ดและโครงสร้าง (Code Quality) โค้ดจัดโครงสร้างเป็นโมดูลสะอาด อ่านง่าย มี Comment อธิบายชัดเจนตามมาตรฐาน Clean Code โค้ดมีการจัดกลุ่มฟังก์ชั่นเป็นระเบียบ มี Comment อธิบายในจุดสำคัญ โค้ดเขียนซ้ำซ้อน ขาดการจัดกลุ่มโมดูล มี Comment น้อย โค้ดอ่านยาก ไม่มีการจัดรูปหน้า ขาด Comment อธิบาย
    4. รายงานเอกสารประกอบ (Documentation) เอกสารสมบูรณ์ มีสถาปัตยกรรมระบบ Diagram, รายละเอียดทดสอบ และผลวิเคราะห์ครบถ้วน เอกสารเรียบร้อย มีเนื้อหาครอบคลุมสถาปัตยกรรมและการใช้งาน เอกสารขาดรายละเอียดบางส่วน ขาดรูปประกอบหรือการวิเคราะห์ผล เอกสารไม่ครบถ้วน ส่งช้ากว่ากำหนดหรือไม่ทำตามรูปแบบ
    5. การนำเสนอและการตอบคำถาม (Presentation) นำเสนอได้กระชับ ชัดเจน สาธิตซอฟต์แวร์ราบรื่น ตอบคำถามเชิงทฤษฎีได้ถูกต้องแม่นยำ นำเสนอเข้าใจง่าย สาธิตระบบได้ดี ตอบคำถามได้ถูกต้องส่วนใหญ่ นำเสนอติดขัดบ้าง สาธิตระบบมีปัญหาเล็กน้อย ตอบคำถามได้บางส่วน นำเสนอไม่ชัดเจน ไม่สามารถสาธิตการทำงานได้ ตอบคำถามไม่ได้
