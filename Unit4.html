วิชา 21909-1012 ระบบฐานข้อมูลสำหรับงาน IoT

หน่วยที่ 4: การสร้างความสัมพันธ์ระหว่างตาราง

เจาะลึกกลไกคีย์หลัก คีย์นอก และการรักษาความถูกต้องของข้อมูล (Referential Integrity) ป้องกันข้อมูลขยะในระบบ IoT

⏱️ เวลาเรียน: 2 สัปดาห์ (สัปดาห์ที่ 8 - 9 | รวมเวลา 8 ชั่วโมง)

🎯 สมรรถนะย่อยประจำหน่วยที่ 4

  • ✅ อธิบายปัญหาความซ้ำซ้อนของข้อมูล และความจำเป็นในการแตกตารางฐานข้อมูลเชิงสัมพันธ์ได้
  • ✅ ใช้คำสั่ง SQL บังคับกฎเกณฑ์คีย์นอก (Foreign Key Constraints) เชื่อมตารางแม่-ลูกได้อย่างรอบคอบ
  • ✅ เลือกใช้เงื่อนไขควบคุมการลบและแก้ไขข้อมูล (CASCADE, RESTRICT) ได้เหมาะสมกับลักษณะงานอาชีพ
4.1

ปัญหาความซ้ำซ้อนของข้อมูล และการแตกตาราง (Data Redundancy)

หากเราบันทึกประวัติตัวอุปกรณ์ ชื่ออาคาร และผู้รับผิดชอบ ปนไปกับค่าที่เซนเซอร์พ่นออกมาทุกๆ 1 นาที ตารางจะบวมและเต็มไปด้วยข้อมูลซ้ำซ้อน สถาปัตยกรรมฐานข้อมูลเชิงสัมพันธ์จึงแก้ปัญหานี้โดยการแยกโครงสร้างออกเป็น "ตารางแม่" และ "ตารางลูก"

❌ แบบตารางเดี่ยว (Flat File) - เกิดผลเสีย

ต้องพิมพ์ข้อความยาวๆ เช่น "เซนเซอร์โซน A เสาต้นที่ 4 นายสมชายดูแล" ซ้ำไปซ้ำมานับล้านแถวตามค่าเซนเซอร์ เปลืองพื้นที่จัดเก็บ และหากเปลี่ยนคนดูแลต้องตามไปแก้ข้อมูลเป็นล้านบรรทัด

✨ แบบแยกตารางสัมพันธ์ (Relational) - ประสิทธิภาพสูง

ตารางแม่: เก็บประวัติอุปกรณ์ครั้งเดียวจบ มี ID กำกับ
ตารางลูก: เก็บเฉพาะค่าที่เซนเซอร์ส่งมา แล้วแปะเลข ID อ้างอิงสั้นๆ กลับมาหาตารางแม่ ข้อมูลจะกระชับและเป็นระบบ

4.2

กลไกคีย์หลัก (Primary Key) และคีย์นอก (Foreign Key)

การเชื่อมสายสัมพันธ์ระหว่างตารางแม่และตารางลูก จะใช้กลไกการผูกคีย์เข้าหากันเพื่อตรวจสอบความถูกต้องสากล (Referential Integrity):

🔑 คีย์หลัก (Primary Key - PK)

คอลัมน์ในตารางแม่ที่ข้อมูลห้ามซ้ำและห้ามว่าง เพื่อใช้ชี้ตัวตนของเซนเซอร์เครื่องนั้นๆ เช่น sensor_id

🔗 คีย์นอก (Foreign Key - FK)

คอลัมน์ที่สร้างขึ้นในตารางลูก เพื่อใช้หยอดค่าเลข ID ที่วิ่งไปชี้เป้ากลับไปหาคีย์หลักในตารางแม่ เปรียบเสมือนสมุดเช็กชื่อรายวันที่จดบันทึกไว้เฉพาะ "รหัสประจำตัวนักเรียน" เท่านั้น

⚠️ กฎเหล็กความถูกต้องของข้อมูล (Referential Integrity)

MySQL จะกางปีกปกป้องระบบทันที โดยระบุว่า ช่องคีย์นอกในตารางลูก จะใส่ตัวเลขที่ไม่มีอยู่จริงในตารางแม่ไม่ได้เด็ดขาด เช่น หากในฟาร์มมีเซนเซอร์แค่เบอร์ 1, 2, 3 แล้วบอร์ด IoT ส่งข้อมูลโดยระบุว่าเป็นของเซนเซอร์เบอร์ 99 เข้ามา ฐานข้อมูลจะตีกลับและบล็อกคำสั่งนี้ทันทีเพื่อป้องกันข้อมูลขยะกำพร้า

4.3

ชุดคำสั่งสร้างความสัมพันธ์ และเงื่อนไขการลบ/แก้ไข (On Delete / On Update)

ในการผูกพฤติกรรมคีย์นอก เราต้องสั่งการระบบล่วงหน้าว่า "ถ้าตารางแม่มีการลบหรือแก้ไขข้อมูล จะให้ตารางลูกทำอย่างไรกับประวัติเก่า?"

🔄 CASCADE (ลบตามกัน/แก้ตามกัน) ถ้ายกเลิกและลบรหัสเซนเซอร์เบอร์ 05 ออกจากตารางแม่ ให้ระบบช่วยลบประวัติค่าอุณหภูมิเก่าๆ ของเซนเซอร์เบอร์นี้ในตารางลูกทิ้งไปให้หมดโดยอัตโนมัติ
🛑 RESTRICT / NO ACTION (บล็อกการลบ) ไม่อนุญาตให้กดลบรหัสเซนเซอร์ออกจากตารางแม่เด็ดขาด ตราบใดที่ในตารางลูกยังมีประวัติข้อมูลค้างอยู่ เพื่อป้องกันความพลั้งเผลอของผู้ดูแลระบบ

-- ตัวอย่างโค้ด SQL สร้างตารางแม่และตารางลูก ผูกความสัมพันธ์แบบ CASCADE

-- 1. สร้างตารางแม่

CREATE TABLE tb_sensors (

sensor_id INT AUTO_INCREMENT PRIMARY KEY,

sensor_mac VARCHAR(50) NOT NULL UNIQUE

);

-- 2. สร้างตารางลูก พร้อมผูกคำสั่ง FOREIGN KEY

CREATE TABLE tb_humidity_logs (

log_id INT AUTO_INCREMENT PRIMARY KEY,

fk_sensor_id INT NOT NULL,

humidity_value FLOAT NOT NULL,

FOREIGN KEY (fk_sensor_id) REFERENCES tb_sensors(sensor_id)

ON UPDATE CASCADE ON DELETE CASCADE

);

🛠️ 2. ลำดับกิจกรรมปฏิบัติจริงในห้องเรียน (รวม 6 ชั่วโมง)

สัปดาห์ที่ 8

ปฏิบัติการปูสายสัมพันธ์โค้ดดิ้ง

นักเรียนลงมือพิมพ์คำสั่งสร้างตารางแม่ tb_sensors และตารางลูกอ้างอิงส่งค่า และสลับบทบาทในการหยอดข้อมูลให้ตรงตามหลักคีย์นอก

สัปดาห์ที่ 9

ทดสอบกฎเหล็กเตือนภัย & Designer View

ฝึกใช้คำสั่ง ALTER TABLE ผูกความสัมพันธ์ย้อนหลัง ทดลองแกล้งป้อนข้อมูลขยะรหัสลอยเพื่อให้ MySQL ตีกลับ และเปิดโหมดลากเส้นสัมพันธ์ส่องดูสถาปัตยกรรมผ่านหน้าจอดีไซเนอร์

📝 ใบงานออนไลน์ที่ 4: ตรวจสอบตรรกะความถูกต้องฐานข้อมูลเชิงสัมพันธ์

ทบทวนความรอบคอบในการควบคุมตารางแม่ตารางลูกผ่านระบบถามตอบอินเทอร์แอคทีฟ

❔ โจทย์ข้อที่ 1: ในตารางลูก คอลัมน์ที่ถูกสร้างขึ้นมาเพื่อเก็บค่ารหัส ID อ้างอิงวิ่งกลับไปหาตารางแม่ เรียกว่าคีย์ประเภทใด?

ตอบ: คีย์นอก หรือ Foreign Key (FK) ซึ่งทำหน้าที่รักษาความถูกต้องของข้อมูลความสัมพันธ์อ้างอิงระหว่างตาราง

❔ โจทย์ข้อที่ 2: หากระบุเงื่อนไขความสัมพันธ์เป็น ON DELETE CASCADE จะเกิดอะไรขึ้นเมื่อเรากดลบรายชื่อเซนเซอร์ในตารางแม่?

ตอบ: ระบบฐานข้อมูลจะตามไป ลบข้อมูลล็อกประวัติเก่าๆ ทั้งหมดในตารางลูกที่ผูกกับเซนเซอร์ตัวนี้ทิ้งให้ทันที โดยอัตโนมัติเป็นทอดๆ

❔ โจทย์ข้อที่ 3: หากบอร์ด IoT พยายามส่งข้อมูลและบันทึกคีย์นอกเป็นรหัสเลขที่ไม่มีอยู่จริงในตารางแม่ ระบบ MySQL จะมีปฏิกิริยาอย่างไร?

ตอบ: ระบบจะขึ้นแจ้งเตือนข้อผิดพลาดข้อความ "Foreign Key Constraint Violation" และทำการบล็อกไม่ยอมให้ข้อมูลชุดนั้นบันทึกลงตารางเด็ดขาดเพื่อป้องกันข้อมูลขยะกำพร้าในระบบ
