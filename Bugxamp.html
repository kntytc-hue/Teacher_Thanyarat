📘 เอกสารและใบความรู้ฉบับสมบูรณ์ประจำหน่วยเรียนรู้

คัมภีร์การวิเคราะห์ระบบและการแก้บั๊ก Apache & MySQL

รวบรวมเหตุการณ์ขัดข้อง บั๊กระบบเครือข่าย โครงสร้างตารางพัง และแนวทางกู้คืนระบบเชิงลึกสำหรับงานไอทีและ IoT

🔍 1. นิยามและโครงสร้างของ Maintenance Log Report

Maintenance Log Report (รายงานบันทึกการซ่อมบำรุงระบบ) คือ คลังความรู้กลางประจำสถานประกอบการที่ช่างเทคนิคคอมพิวเตอร์และวิศวกรระบบใช้บันทึกเหตุการณ์เมื่อระบบเกิดข้อผิดพลาด (Error) หรือบั๊กระบบ (System Bug) เพื่อเปลี่ยนข้อมูลความเสียหายในอดีต ให้กลายเป็น "คู่มือแก้ปัญหาในอนาคต" ป้องกันไม่ให้ทีมงานต้องเสียเวลาคลำหาทางแก้ไขใหม่เมื่อเกิดปัญหาเดิมซ้ำซาก

💡 การเปรียบเทียบให้นักเรียนเห็นภาพชัดเจน: "ประวัติการรักษาของแพทย์"

ให้นักเรียนจินตนาการว่า XAMPP หรือระบบฐานข้อมูลคือ "คนไข้" เมื่อระบบเปิดไม่ติดหรือไฟแดงขึ้น ก็เหมือนคนไข้ป่วยหนักเดินมาหาหมอ ช่างเทคนิคคือ "คุณหมอ" และข้อความภาษาอังกฤษในกล่องดำ (Console Log) ก็คือ "ผลตรวจเลือดหรือภาพ X-Ray"

หมอที่ดีจะไม่เดาสุ่มผ่าตัดทันที แต่ต้องอ่านผลตรวจเลือด (Log) ให้ละเอียด วินิจฉัยให้ตรงโรค แล้วเขียนบันทึกรายงานการรักษา (Maintenance Log Report) ว่าใช้ยาอะไรคนไข้ถึงรอดชีวิต เพื่อให้หมอคนต่อไปรักษาต่อได้ถูกต้อง!

🛠️ 2. ทบทวน 5 ปัญหาคลาสสิกของ XAMPP & MySQL ในงาน IoT

CASE 1: Port 3306 Conflict (พอร์ต MySQL ชนกับโปรแกรมอื่น)

[MySQL] Error: Port 3306 is occupied by another application PID 404! Exiting...

สาเหตุ: โปรแกรมฐานข้อมูลอื่น หรือซอฟต์แวร์จำลองเครือข่ายแอบเปิดรันและยึดพอร์ตสากล 3306 ไว้ก่อนแล้ว ทำให้ MySQL ของ XAMPP เปิดไม่ขึ้น (ขึ้นไฟแดง/ดับทันที)
มาตรการแก้ไข: คลิกปุ่ม Config ที่ MySQL ➔ เลือกไฟล์ my.ini ➔ แก้ไขจาก port=3306 เป็น port=3307 จากนั้นบันทึกไฟล์แล้วกด Start ระบบใหม่

CASE 2: Missing Database Connection URL (พิมพ์ URL เข้าฐานข้อมูลผิด)

[Apache Server] HTTP Error 404: Requested URL /phpadmin was not found on this server.

สาเหตุ: พิมพ์เส้นทางเข้าสู่หน้าต่างกราฟิกจัดการฐานข้อมูลบน Browser ผิดพลาด ทำให้ตัวเว็บเซิร์ฟเวอร์ตอบกลับเป็นรหัสพัง 404 Not Found
มาตรการแก้ไข: ตรวจสอบตัวสะกดใหม่ พิมพ์ที่อยู่ URL ให้ถูกต้องตรงตามมาตรฐานคือ localhost/phpmyadmin หรือ 127.0.0.1/phpmyadmin

CASE 3: Apache Port 80 Blocked (พอร์ตเว็บเซิร์ฟเวอร์โดนบล็อก)

[Apache] Error: Port 80 is locked by PID 1420 (Skype.exe). Binding failed!

สาเหตุ: โปรแกรมสื่อสารหรือบริการ IIS ของ Windows แอบดักใช้งานพอร์ตเว็บเบื้องต้น (Port 80) ทำให้รันตัว Apache Server ไม่ได้
มาตรการแก้ไข: คลิกปุ่ม Config ที่ Apache ➔ เลือกไฟล์ httpd.conf ➔ แก้ไขจาก Listen 80 เป็น Listen 8080 จากนั้นเข้าผ่าน localhost:8080 แทน

CASE 4: Root Password Mismatch (รหัสผ่านบอร์ด IoT ไม่ตรงกับ MySQL)

[MySQL Server] Access denied for user 'root'@'localhost' (using password: YES)

สาเหตุ: ตัวอุปกรณ์เซนเซอร์ปลายทาง พยายามส่งรหัสผ่านเพื่อขอเปิดสิทธิ์บันทึกข้อมูล แต่รหัสในโค้ดฝั่งฮาร์ดแวร์ไม่ตรงกับที่ฐานข้อมูลส่วนกลางตั้งค่าไว้
มาตรการแก้ไข: แก้ไขโค้ดการเชื่อมต่อบนบอร์ด IoT ปลายทาง ตรวจสอบฟิลด์ password ให้ตรงกับค่าเริ่มต้นของ XAMPP MySQL คือปล่อยเป็นช่องว่างเปล่า ""

CASE 5: Forgot to Start MySQL (ลืมกดปุ่ม Start ตัว MySQL Service)

[PHP/IoT Engine] Fatal Warning: mysqli::real_connect(): Connection refused by host machine.

สาเหตุ: อุปกรณ์ฮาร์ดแวร์พยายามยิงข้อมูลเข้ามา แต่ฝั่งตัวรับหรือเซิร์ฟเวอร์ไม่ได้เปิดเครื่องยนต์รับข้อมูลเอาไว้ เพราะลืมกดปุ่มเปิดบริการใน XAMPP Control Panel
มาตรการแก้ไข: เปิดหน้าต่าง XAMPP Control Panel ขึ้นมา แล้วตรวจเช็กความรอบคอบโดยการคลิกกดปุ่ม Start หน้าบริการ MySQL เพื่อให้ออนไลน์

💥 3. คลังสารพัดบั๊กและข้อผิดพลาดร้ายแรงขั้นสูงของ Apache & MySQL

รวบรวมเหตุการณ์บั๊กของระบบหน่วยความจำ สิทธิ์ความปลอดภัย และโครงสร้างข้อมูลพังทลายในระดับวิศวกรรมเครือข่าย:

CASE 6: Table Corrupted / Crashed (โครงสร้างตารางข้อมูลใน MySQL พังทลาย)

[ERROR] MySQL: Table '.\smart_farm_db\tb_logs' is marked as crashed and should be repaired

อาการและสาเหตุ: บอร์ด IoT กำลังยิงข้อมูลเข้าตารางอยู่ดีๆ แล้วเกิดเหตุไฟดับกะพริบ หรือฮาร์ดดิสก์ของเซิร์ฟเวอร์เต็มกะทันหัน ทำให้ดัชนีของตารางเสียหาย ส่งผลให้เรียกดูข้อมูลหรือเขียนข้อมูลใหม่ไม่ได้เด็ดขาด
มาตรการแก้ไข: คลิกปุ่ม Shell ที่แถบเมนูด้านขวาของ XAMPP เพื่อเปิดหน้าต่างเทอร์มินัลสีดำ จากนั้นพิมพ์คำสั่งกู้คืนโครงสร้างตารางย้อนหลังดังนี้:

mysqlcheck -u root --repair --all-databases

CASE 7: Too Many Connections (จำนวนการเชื่อมต่อเซนเซอร์เกินขีดจำกัด)

Fatal error: Uncaught mysqli_sql_exception: Too many connections in server node.

อาการและสาเหตุ: เกิดจากการขยายระบบ IoT มีบอร์ดเซนเซอร์จำนวนมาก (เช่น 200-300 ตัว) ยิงข้อมูลเข้ามาพร้อมกัน โดยในโค้ดไม่ได้เขียนคำสั่งปิดการเชื่อมต่อเมื่อส่งเสร็จ ทำให้คิวการเชื่อมต่อเต็มขีดจำกัดสูงสุดที่ MySQL ตั้งไว้ (ค่าเริ่มต้น XAMPP มักจำกัดไว้ที่ 151 คิว)
มาตรการแก้ไข: เข้าไปที่ไฟล์ my.ini ของ MySQL ค้นหาหมวดหมู่นี้แล้วทำการเพิ่มขีดจำกัดความจุคิวตัวแปรให้รองรับบอร์ดเพิ่มขึ้น:

max_connections = 1000

CASE 8: Remote Access Blocked (ปฏิเสธสิทธิ์การเชื่อมต่อจากไอพีภายนอก)

[Warning] MySQL: Connection from remote host '192.168.1.50' is not allowed by security rules.

อาการและสาเหตุ: บอร์ด ESP32 ต่อเน็ตวงเดียวกับคอมพิวเตอร์และยิงไอพีเครื่องเซิร์ฟเวอร์ตรงๆ แต่ฐานข้อมูลไม่ยอมรับข้อมูล เนื่องจากระบบความปลอดภัยเริ่มต้นของ XAMPP จะอนุญาตให้เฉพาะเครื่องตัวเอง (localhost / 127.0.0.1) ใช้งานได้เท่านั้น
มาตรการแก้ไข: เปิดหน้าต่าง phpMyAdmin เข้าไปที่เมนู User accounts ➔ เลือกบัญชีผู้ใช้ root ➔ เปลี่ยนจาก localhost ให้กลายเป็นเครื่องหมายเปอร์เซ็นต์ % (หมายถึงอนุญาตให้ไอพีเครือข่ายภายนอกวิ่งเข้ามาคุยได้ทั้งหมด)

CASE 9: Max Execution Time Exceeded (Apache ตัดการทำงานเพราะประมวลผลนานเกินไป)

Fatal Error: Maximum execution time of 30 seconds exceeded in query_report.php

อาการและสาเหตุ: เมื่อฐานข้อมูล IoT สะสมค่าเซนเซอร์มาหลายล้านแถว ช่างเทคนิคกดเขียนโค้ดดึงข้อมูลทั้งหมดมาพลอตกราฟย้อนหลัง หน้าเว็บหมุนค้าง 30 วินาทีแล้วตัดเป็นหน้าจอพัง เพราะระบบ Apache มีบัฟเฟอร์ตัดเวลาป้องกันเว็บค้างค้างไว้
มาตรการแก้ไข: คลิกปุ่ม Config ของ Apache ➔ เลือกไฟล์ php.ini ➔ ค้นหาและปรับขยายระยะเวลาหน่วงวินาทีเพิ่มขึ้นเพื่อให้เซิร์ฟเวอร์วิเคราะห์ข้อมูลก้อนใหญ่ได้สำเร็จ:

max_execution_time = 300 ; ขยายเวลาเป็น 5 นาที

CASE 10: InnoDb Database Engine Corruption (ไฟล์ระบบภายในคลังล่มสลายถาวร)

[InnoDB] Fatal error: Cannot open a tablespace because tablespace id in file .\ibdata1 mismatch.

อาการและสาเหตุ: บั๊กขั้นวิกฤตสูงสุด เกิดจากโครงสร้างหลักของ InnoDB เสียหายรุนแรงจากการปิดคอมพิวเตอร์กะทันหัน ส่งผลให้ปุ่ม MySQL ขึ้นสีแดงถาวรและไม่สามารถกด Start ได้เลยแม้จะแก้พอร์ตแล้วก็ตาม
มาตรการแก้ไขอย่างรอบคอบ: ห้ามลบโปรแกรมทิ้งเด็ดขาด! ให้เข้าไปที่ไฟล์ my.ini ค้นหาตำแหน่งหัวข้อ [mysqld] แล้วเติมบรรทัดคำสั่งบังคับกู้ชีพไฟล์ระบบชั่วคราวลงไป:

innodb_force_recovery = 1

*(คำแนะนำ: เมื่อเปิดระบบติดแล้ว ให้รีบทำการ Export ข้อมูลทั้งหมดออกมาเก็บไว้ จากนั้นค่อยลบตารางและปรับค่า recovery กลับเป็น 0 ตามเดิม)

📊 4. โครงสร้างฟิลด์มาตรฐานสำหรับเขียนบันทึกรายงาน Maintenance Log

หัวข้อรายงาน เจตจำนงในการบันทึก ตัวอย่างร่องรอยคีย์เวิร์ดในระบบ (Log Keyword)
1. Issue Description บันทึกอาการภายนอกที่มองเห็นของตัวบอร์ดฮาร์ดแวร์ หรือความล่าช้าของหน้าเว็บ Dashboard "บอร์ดหลุดการเชื่อมต่อ", "หน้าต่าง XAMPP ดีดดับขึ้นไฟแดง", "กราฟหมุนค้างยาวนานแล้วขึ้น 404"
2. Error Log Evidence คัดลอกข้อความแจ้งเตือนภาษาอังกฤษดั้งเดิมในช่องดำมาวางไว้ ห้ามแก้ไขข้อความเด็ดขาด "Too many connections", "Table is marked as crashed", "Access denied for user root"
3. Root Cause Analysis แปลไทยและวิเคราะห์สรุปสาเหตุต้นตอเชิงเครือข่ายและหน่วยความจำให้หัวหน้างานทราบ "คิวเต็มเนื่องจากต่อบอร์ดเซนเซอร์มากเกินไป", "ไฟดับทำโครงสร้างตารางเจ๊ง", "พอร์ต 80 ชนกับ Skype"
4. Corrective Action บันทึกขั้นตอนปฏิบัติต่างๆ ที่ช่างใช้ผ่าตัดซ่อมบำรุงไฟล์ระบบจนระบบคืนชีพอย่างปลอดภัย "เพิ่มค่า max_connections=1000 ใน my.ini", "รัน mysqlcheck ซ่อมโครงสร้างผ่าน Shell"
